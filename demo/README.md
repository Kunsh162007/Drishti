## What is DRISHTI?

**DRISHTI** turns raw FIR records into actionable intelligence. It ingests crime, person and vehicle data and surfaces **where** crime concentrates (statistical hotspots), **what's emerging** (near‑repeat / intensifying clusters), **who's connected** (person–vehicle–crime networks and gang communities), **which records are anomalous**, **which identities are the same person** under spelling variants (entity resolution), and **which cases share a modus operandi**. A **grounded assistant** answers natural‑language questions using *only* retrieved records — it cites the FIRs it used and never fabricates. Everything in this demo runs on free, open tooling with a bundled synthetic Karnataka dataset, so you can click "deploy" and have a live URL in minutes.

> The demo is honest by design: missing fields are **reported, never invented**, and the assistant **refuses** when it has no records to cite.

## 🚀 Quick start (local)

> Run everything **from inside the `demo/` folder**. The data scripts work from any directory (they locate the repo root from their own file path), but uvicorn needs `backend.main:app` importable — which only works from inside `demo/`.

### macOS / Linux / Git‑Bash

```bash
cd demo
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
python scripts/generate_data.py     # writes data/seed/*.csv + data/samples/*
python scripts/seed_db.py           # builds data/drishti_demo.db (SQLite)
uvicorn backend.main:app --reload   # open http://localhost:8000
```

Or just use the convenience runner: `./start.sh` (creates the venv, installs, seeds, serves).

### Windows (PowerShell) — exact commands

```powershell
cd demo
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
python scripts\generate_data.py
python scripts\seed_db.py
uvicorn backend.main:app --reload
# then open http://localhost:8000
```

Then visit **http://localhost:8000** (frontend) and **http://localhost:8000/docs** (interactive API). Health check: **http://localhost:8000/api/health**.

> **No external database needed.** The default DB is a bundled SQLite file at `demo/data/drishti_demo.db`, generated by the two scripts above. The assistant is free by default (`LLM_PROVIDER=none`) and gives deterministic, grounded answers — no API key required.

## ☁️ Deploy free on Render

This repo ships a **Render Blueprint** (`demo/render.yaml`) for the **free** plan — no database to provision, no API key.

1. **Push to GitHub.** Make sure the `demo/` folder is at the repo root.
2. In the **Render Dashboard** → **New** → **Blueprint**.
3. **Connect your repository.** Render detects `demo/render.yaml` automatically (point it there if prompted).
4. **Apply.** The first build runs:
   ```
   pip install -r requirements.txt
   python scripts/generate_data.py
   python scripts/seed_db.py
   ```
   and then starts `uvicorn backend.main:app --host 0.0.0.0 --port $PORT`. Build takes ~1–2 minutes.
5. Open the generated `*.onrender.com` URL. Health check is at `/api/health`.

> ⚠️ **Free‑tier cold start:** Render spins the free web service down after ~15 minutes of inactivity. The **first** request after idle takes ~30–60 s to wake the service (and re‑load SQLite into memory). Subsequent requests are fast. This is a Render free‑plan trait, not a DRISHTI limitation — a paid instance type removes it.

## 📥 Connect your own dataset

You don't have to use the synthetic data. Push a file straight into a running instance via the ingest endpoint.

**Endpoint:** `POST /api/ingest` — multipart form with:
- `file`: your data as **CSV, JSON, NDJSON, GeoJSON, or XLSX**.
- `mapping` *(optional)*: a JSON object mapping **your** column names → DRISHTI's **canonical** fields.

Two helper files are generated into `demo/data/samples/`:
- **`connect_your_own_TEMPLATE.csv`** — a header‑only CSV with every canonical column, so you can see exactly what DRISHTI expects.
- **`MAPPING_EXAMPLE.json`** — a worked example mapping arbitrary headers (e.g. `"FIR No" → fir_number`, `"Offence" → crime_type`, `"Loss Value" → property_value_inr`).

```bash
curl -X POST http://localhost:8000/api/ingest \
  -F "file=@data/samples/new_incidents.csv" \
  -F 'mapping={"FIR No":"fir_number","Offence":"crime_type","Lat":"latitude","Long":"longitude"}'
```

Response shape:
```json
{ "inserted": 593, "skipped": 7, "missing_report": { "weapon_used": 412, "phone": 88 }, "errors": [] }
```

> **Honest ingestion.** Date columns are parsed to ISO‑8601; `hour`, `day_of_week`, and `h3_*` are *derived* when lat/lng are present. Any canonical field you don't map is counted in `missing_report` — **never back‑filled or fabricated**.

## 🏭 Going to production (paid, full power)

**The headline:** it's the **same codebase and the same features**. Production differs *only* by environment variables and the infrastructure they point at. Nothing is rewritten — you change config, not code.

| Component | Demo (free) | Production swap | How (env / infra) |
|---|---|---|---|
| **Database** | SQLite file (`sqlite:///…drishti_demo.db`) | **PostgreSQL + PostGIS** | `DATABASE_URL=postgresql+psycopg://…/drishti` |
| **Geospatial** | H3 indexing in app | PostGIS spatial types + H3 in DB | same code; queries pushed down to PostGIS |
| **Assistant** | Extractive, grounded (`LLM_PROVIDER=none`) | **Claude** | `LLM_PROVIDER=anthropic`, `LLM_API_KEY=…`, `LLM_MODEL=claude-opus-4-8` |
| **Assistant (sovereign)** | — | **Self‑hosted Llama** (on‑prem GPU) | `LLM_PROVIDER=groq`/self‑host, `LLM_MODEL=llama-3.3-70b` |
| **Graph / networks** | NetworkX in‑process | **Neo4j** graph DB | network/community code reads from Neo4j; same endpoints |
| **Semantic search / RAG** | keyword + rapidfuzz | **Qdrant** vector store | embeddings for MO‑linkage & assistant retrieval |
| **Secrets / keys** | env vars | **HSM / KMS** (GovCloud) | keys never on disk; sealed by hardware module |
| **Hosting** | Render free web service | On‑prem / GovCloud, autoscaled, behind reverse proxy | container is identical |

In every row the *application* is unchanged — DRISHTI reads `DATABASE_URL`, `LLM_PROVIDER`, `LLM_API_KEY`, `LLM_MODEL` at startup and behaves accordingly. See **`../production/`** for a realistic docker‑compose topology (app + Postgres/PostGIS + Neo4j + Qdrant + reverse proxy) and the full production env template.

### Environment variables (the demo↔prod switch)

| Var | Demo default | Production example |
|---|---|---|
| `DRISHTI_MODE` | `demo` | `production` |
| `DATABASE_URL` | `sqlite:///…/drishti_demo.db` | `postgresql+psycopg://…/drishti` (PostGIS) |
| `LLM_PROVIDER` | `none` | `anthropic` / `groq` / `openai` |
| `LLM_API_KEY` | *(empty)* | *(paid / sovereign key)* |
| `LLM_MODEL` | *(empty)* | `claude-opus-4-8` / `llama-3.3-70b` |
| `PORT` | `8000` | `8000` (behind proxy) |

## 🔎 What's simplified in the demo vs the full proposal

This demo is faithful to the architecture but deliberately scoped to stay **free and instantly deployable**. In the interest of honesty:

- **Data is synthetic.** ~8,000 generated Karnataka FIRs (+50k stress set) — realistic patterns, but not real case data.
- **One process, one SQLite file.** No PostGIS, Neo4j, or Qdrant running; their roles are emulated in‑process (NetworkX for graphs, rapidfuzz/keyword for similarity). Production splits these into dedicated services (see `../production/`).
- **Assistant is extractive by default.** Grounded and cited, but not an LLM until you set `LLM_PROVIDER`. This keeps the demo key‑free and 100% deterministic.
- **MyShield uses a demo token.** Production wires real citizen auth (OTP / DigiLocker‑style) and access control.
- **No HSM/KMS, no on‑prem sovereignty controls, no audit pipeline** in the demo — these are production concerns covered in the proposal and the `../production/` topology.
- **Analytics are tuned for clarity.** Thresholds and parameters are set so each view "lights up" on the synthetic data; production calibrates against real distributions.

Everything else — the API contract, the analytics methods, the grounded‑assistant discipline, the ingestion honesty — is the real thing.

## 📚 API reference

All endpoints are JSON under the `/api` prefix (see `SPEC.md` for full request/response schemas).

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/api/health` | `{status, mode, db, records}` |
| `GET` | `/api/meta` | Districts, crime types, categories, date range, totals |
| `GET` | `/api/crimes` | Filtered crime list (`district`, `crime_type`, `category`, `date_from/to`, `q`, `limit`, `offset`) |
| `GET` | `/api/stats` | KPIs + breakdowns by category / district / hour / month / status |
| `GET` | `/api/hotspots` | Getis‑Ord Gi\* hotspots on H3 cells (`resolution` 7–9) |
| `GET` | `/api/emerging` | New / intensifying / persistent / diminishing cells |
| `GET` | `/api/timeseries` | Counts per `day`/`week`/`month` |
| `GET` | `/api/anomalies` | Outlier FIRs with reasons |
| `GET` | `/api/risk` | Explainable place‑based risk (0–1) with drivers |
| `GET` | `/api/network` | Person/vehicle/crime/station graph (`fir`, `person`, `depth`, `limit`) |
| `GET` | `/api/network/communities` | Detected communities / gang components |
| `GET` | `/api/entity-resolution` | Same‑person pairs with evidence + decision |
| `GET` | `/api/mo-linkage` | Cases sharing modus operandi (`fir`, `top_k`) |
| `POST` | `/api/assistant/chat` | Grounded NL answer + citations + map filter |
| `POST` | `/api/ingest` | Upload CSV/JSON/NDJSON/GeoJSON/XLSX (+ optional `mapping`) |
| `GET` | `/api/myshield` | Citizen view: only your own records + de‑identified area aggregates |

Interactive docs are served at **`/docs`** on any running instance.
